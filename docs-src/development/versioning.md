# Versioning System

This guide explains how versioning works for the Customized Gradebook project, including semantic versioning, version tracks, and the auto-patch system.

## Semantic Versioning

Both the main CG and mobile module use **semantic versioning** (semver):

```
v{major}.{minor}.{patch}
```

### Version Components

- **Major** - Breaking changes, incompatible API changes
- **Minor** - New features, backward-compatible
- **Patch** - Bug fixes, backward-compatible

### Examples

| Version | Type | Description |
|---------|------|-------------|
| `v1.2.0 → v1.2.1` | Patch | Bug fix |
| `v1.2.0 → v1.3.0` | Minor | New feature |
| `v1.2.0 → v2.0.0` | Major | Breaking change |

## Independent Versioning

The main CG and mobile module have **independent version numbers**:

- **Main CG:** `v1.2.0` (started at v1.0.0)
- **Mobile Module:** `mobile-v0.1.1` (started at v0.1.0)

This allows each component to evolve at its own pace.

## Version Tracks

The `versions.json` file provides **version tracks** for auto-patch loaders:

```json
{
  "v1.2-latest": "v1.2.0",
  "v1.1-latest": "v1.1.0",
  "v1.0-latest": "v1.0.11",
  "latest": "v1.2.0",
  "stable": "v1.2.0"
}
```

### Track Types

| Track | Description | Example |
|-------|-------------|---------|
| `latest` | Newest version overall | `v1.2.0` |
| `stable` | Current stable version | `v1.2.0` |
| `v1.2-latest` | Latest patch in v1.2.x series | `v1.2.0` |
| `v1.1-latest` | Latest patch in v1.1.x series | `v1.1.0` |

### How Tracks Work

When you release a new version:

1. **Patch release** (v1.2.0 → v1.2.1):
   - Updates `v1.2-latest` to `v1.2.1`
   - Updates `latest` to `v1.2.1`
   - Updates `stable` to `v1.2.1`

2. **Minor release** (v1.2.1 → v1.3.0):
   - Creates new track `v1.3-latest` → `v1.3.0`
   - Updates `latest` to `v1.3.0`
   - Updates `stable` to `v1.3.0`
   - Keeps `v1.2-latest` at `v1.2.1` (for users who want to stay on v1.2.x)

3. **Major release** (v1.3.0 → v2.0.0):
   - Creates new track `v2.0-latest` → `v2.0.0`
   - Updates `latest` to `v2.0.0`
   - May keep `stable` at `v1.3.0` until v2.0 is proven stable

## Auto-Patch System

The auto-patch system allows loaders to automatically get the latest patch version without manual updates.

### How It Works

**Loader code:**
```javascript
const VERSION_TRACK = "v1.2-latest";

fetch("https://morenet-canvas.github.io/CustomizedGradebook/versions.json")
    .then(r => r.json())
    .then(manifest => {
        const version = manifest[VERSION_TRACK]; // Resolves to "v1.2.1"
        // Load that version from GitHub Releases
    });
```

**Benefits:**
- Admins get automatic patch updates (bug fixes)
- No need to edit loader code for patches
- Can pin to specific minor version (e.g., stay on v1.2.x)

### Version Track Strategy

**For admins who want:**

- **Latest everything:** Use `"latest"` track
- **Stable only:** Use `"stable"` track
- **Pin to minor version:** Use `"v1.2-latest"` track
- **Pin to exact version:** Use `"v1.2.0"` (no track)

## Release Tags

### Main CG Tags

Format: `v{major}.{minor}.{patch}`

Examples:
- `v1.0.0`
- `v1.2.0`
- `v1.2.1`

### Mobile Module Tags

Format: `mobile-v{major}.{minor}.{patch}`

Examples:
- `mobile-v0.1.0`
- `mobile-v0.1.1`
- `mobile-v0.2.0`

### Special Tags

| Tag | Description |
|-----|-------------|
| `dev` | Rolling development build (main CG) |
| `mobile-dev` | Rolling development build (mobile) |

## versions.json Generation

The `versions.json` file is auto-generated by the `update-version-manifest.yml` workflow:

**Process:**
1. Scans all git tags matching `v*` and `mobile-v*`
2. Groups by major.minor version
3. Finds latest patch for each group
4. Generates tracks (`latest`, `stable`, `v1.2-latest`, etc.)
5. Deploys to GitHub Pages

**Triggered by:**
- Release scripts (auto-triggered)
- Manual workflow dispatch

## Best Practices

### For Developers

- **Patch releases** - Bug fixes only, no new features
- **Minor releases** - New features, maintain backward compatibility
- **Major releases** - Breaking changes, update documentation

### For Admins

- **Production:** Use `stable` track or pin to specific minor version
- **Testing:** Use `latest` track or `dev` build
- **Conservative:** Pin to exact version (e.g., `v1.2.0`)

## Future: Mobile Version Tracks

Currently, mobile versions are not included in `versions.json`. Future enhancement:

```json
{
  "latest": "v1.2.0",
  "mobile-latest": "mobile-v0.1.1",
  "mobile-0.1-latest": "mobile-v0.1.1"
}
```

This would enable auto-patch loaders for mobile module.
